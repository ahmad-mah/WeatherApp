# Flutter Deployment Workflow

## Base Workflow
This workflow covers the steps to prepare, build, sign, and deploy a Flutter application to production (App Store, Play Store, Web, or Desktop).

---

## Deployment Flow

```
1. Pre-flight Checks → 2. Versioning → 3. Build & Sign → 4. Test Release Build → 5. Distribute
```

---

## Step 1: Pre-flight Checks

Before building a release, ensure the codebase is clean and optimized.

### Actions

1. **Run Flutter Analyze:**
   ```bash
   flutter analyze
   ```
   *Must return "No issues found!"*

2. **Run Full Test Suite:**
   ```bash
   flutter test
   ```
   *All tests must pass.*

3. **Check Dependencies:**
   ```bash
   flutter pub outdated
   ```
   *Consider updating dependencies if major versions are behind, but DO NOT update right before a release unless necessary for the release.*

4. **Clean the Build Cache:**
   ```bash
   flutter clean
   flutter pub get
   ```

---

## Step 2: Versioning

Flutter uses `pubspec.yaml` for versioning.

Format: `version: major.minor.patch+build_number`
Example: `version: 1.2.3+14`

### Actions

1. **Increment the Version:**
   - **Major:** Breaking changes, massive new features (e.g., 1.x.x to 2.0.0)
   - **Minor:** New features, non-breaking (e.g., 1.2.x to 1.3.0)
   - **Patch:** Bug fixes (e.g., 1.2.3 to 1.2.4)
2. **Increment the Build Number:**
   - The number after `+` MUST be incremented for every submission to the App Store or Play Store. (e.g., `+14` to `+15`).

3. **Update Changelog:**
   Document user-facing changes in `CHANGELOG.md` or your release notes document.

---

## Step 3: Build & Sign

Release builds are compiled ahead-of-time (AOT) for maximum performance and have tree-shaking applied.

### Android Deployment (Google Play)

1. **Configure Signing:** Ensure `key.properties` and `build.gradle` are configured for release signing.
2. **Build App Bundle (AAB):**
   ```bash
   flutter build appbundle --release --obfuscate --split-debug-info=build/app/outputs/symbols
   ```
   *Note: Always use `--obfuscate` to protect your code and `--split-debug-info` to save symbol files for crash reporting.*

### iOS Deployment (App Store)

1. **Update CocoaPods:**
   ```bash
   cd ios && pod install && cd ..
   ```
2. **Configure Signing in Xcode:** Open `ios/Runner.xcworkspace` and ensure the correct Team and Provisioning Profile are selected for the Release build configuration.
3. **Build IPA:**
   ```bash
   flutter build ipa --release --obfuscate --split-debug-info=build/ios/outputs/symbols
   ```

### Web Deployment

1. **Build Web App:**
   ```bash
   flutter build web --release --web-renderer canvaskit
   ```
   *Note: Choose the renderer (`canvaskit` or `html`/`auto`) based on your app's needs.*

---

## Step 4: Test Release Build

**Crucial:** Release builds strip out debugging code and assertions. Code that works in debug mode might fail in release mode (e.g., missing assets, obfuscation breaking reflection).

### Actions

1. **Install Android Release Build:**
   ```bash
   flutter install --release
   ```
   Or build the APK and install it: `flutter build apk --release`

2. **Run iOS Release Build:**
   Run from Xcode using the Release scheme, or use TestFlight.

3. **Verification Checklist on Release Build:**
   - [ ] App launches successfully (no white screen of death)
   - [ ] Authentication flows work (often affected by obfuscation or missing production keys)
   - [ ] Network requests succeed (ensure internet permissions are in AndroidManifest)
   - [ ] Animations and scrolling are smooth
   - [ ] Crash reporting (e.g., Firebase/Sentry) successfully captures a forced crash

---

## Step 5: Distribute

### CI/CD Integration (Recommended)

Manual builds are prone to error. Use CI/CD (e.g., Fastlane, GitHub Actions, Codemagic, Bitrise).

**Standard CI/CD Pipeline:**
1. Trigger on merge to `main` or tag creation
2. Run `flutter analyze` and `flutter test`
3. Increment build number automatically
4. Build `--release --obfuscate`
5. Upload symbols to crash reporting tool (Sentry, Firebase Crashlytics)
6. Publish to TestFlight (iOS) and Play Console Internal Track (Android)

### Managing Obfuscation Symbols

When you use `--obfuscate`, your stack traces will be unreadable. You MUST upload the symbols generated by `--split-debug-info` to your crash reporting service to de-obfuscate stack traces in production.

## Common Deployment Mistakes

| Mistake | Consequence | Fix |
| :--- | :--- | :--- |
| Forgetting to increment build number | Store rejects the upload | Always increment `+build_number` |
| Not testing the `--release` build locally | App crashes on launch for real users | Install and test release APK/IPA before publishing |
| Leaving debug keys/URLs in production | Security risk, app hits staging DB | Use environment variables (e.g., `--dart-define`) |
| Losing the Android keystore file | Cannot update the app on Play Store EVER again | Back up the keystore securely (password manager, secure vault) |
| Not uploading debug symbols | Production crash logs are unreadable | Integrate symbol upload into your CI/CD script |

---

## Quality Checklist

- [ ] `flutter analyze` and `flutter test` pass
- [ ] Version and build number are incremented in `pubspec.yaml`
- [ ] Environment variables point to PRODUCTION (API keys, URLs)
- [ ] App built with `--release` flag
- [ ] App built with `--obfuscate` and `--split-debug-info`
- [ ] Release build tested manually on a real device
- [ ] Debug symbols uploaded to crash reporting tool
- [ ] Release notes written
